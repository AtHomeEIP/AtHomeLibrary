28 /*
29  * Thinking communication wisely
30  * - Shall we have only two communication interface? Sync + WoodBox? Two ICommunicators with dedicated methods?
31  * - Shall we have a variable number of interchangeable communication interface?
32  * - Shall we have an interface of already defined methods to implement?
33  * - Shall we have a dictionary of methods to use when receiving a message?
34  *
35  * Communication protocol:
36  * - End of line: \r\n string
37  * - Spacer: \t
38  * - End of command: end of text character
39  * - End of communication: end of transmission character
40  *
41  * - Line 1 should be method name
42  * - There should be an array of parameters available to use for requests
43  * - Request body should be able to transmit on a character mode interface, thus using encoding such as Base64
44  *
45  * Communication protocol format:
46  * [aZ-09]+[EOL] // First line should be method name, insensible to format
47  * ([aZ-09][Spacer]*)+[EOL] // N following lines is parameters array
48  * [Separator][EOL] // Indicates end of parameters and beginning of body
49  * [Stream][EOC][EOT]
50  *
51  * Communication methods
52  * - Enumerate: Give module information, describe itself. Maybe should have a better name
53  * - UploadData: Module sensor samples sent over interface. How to chose which interface if we don't specialize them? Configuration?
54  * - SyncTime: Update clock of module
55  * - SetWiFi: Set module WiFi configuration. How does a module know a ICommunicator is a WiFi interface?
56  * - SetEndPoint: Set TCP socket endpoint (ip + port). How does a module know a ICommunicator implements a TCP/IP stack?
57  * - SetProfile: Set a configuration profile to the module to store internally. What is supposed to be inside? What the module is supposed to do with it?
58  *
59  * Request object
60  * - Should be derived from Printable?
61  * - Attributes
62  * - Method name -> defined by constructor? Setter? Builder class?
63  * - Parameters -> defined by dictionary? Attributes in a class?
64  * - Body -> Blob
65  *
66  * Proposition:
67  * - ICommunicators classes are discovered through dynamic_cast, eg.: A SetWiFi command is received,
68  * the corresponding receiver method iterate through all ICommunicators and if some are AWiFiCommunicator,
69  * they update the settings. But this exclude having several communicators of same type connected to several different pairs
70  * - Event receivers are specialized classes implementing an interface, and are associated in a map to the corresponding event
71  * - SendingCommand objects are a helper class used to send a command. Predefined command could have specialized classes implementing their action
72  * */
73  }
