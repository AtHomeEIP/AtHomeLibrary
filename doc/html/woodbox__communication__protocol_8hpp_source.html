50 /*
51  * Thinking communication wisely
52  * - Shall we have only two communication interface? Sync + WoodBox? Two ICommunicators with dedicated methods?
53  * - Shall we have a variable number of interchangeable communication interface?
54  * - Shall we have an interface of already defined methods to implement?
55  * - Shall we have a dictionary of methods to use when receiving a message?
56  *
57  * Communication protocol:
58  * - End of line: \r\n string
59  * - Spacer: \t
60  * - End of command: end of text character
61  * - End of communication: end of transmission character
62  *
63  * - Line 1 should be method name
64  * - There should be an array of parameters available to use for requests
65  * - Request body should be able to transmit on a character mode interface, thus using encoding such as Base64
66  *
67  * Communication protocol format:
68  * [aZ-09]+[EOL] // First line should be method name, insensible to format
69  * ([aZ-09][Spacer]*)+[EOL] // N following lines is parameters array
70  * [Separator][EOL] // Indicates end of parameters and beginning of body
71  * [Stream][EOC][EOT]
72  *
73  * Communication methods
74  * - Enumerate: Give module information, describe itself. Maybe should have a better name
75  * - UploadData: Module sensor samples sent over interface. How to chose which interface if we don't specialize them? Configuration?
76  * - SyncTime: Update clock of module
77  * - SetWiFi: Set module WiFi configuration. How does a module know a ICommunicator is a WiFi interface?
78  * - SetEndPoint: Set TCP socket endpoint (ip + port). How does a module know a ICommunicator implements a TCP/IP stack?
79  * - SetProfile: Set a configuration profile to the module to store internally. What is supposed to be inside? What the module is supposed to do with it?
80  *
81  * Request object
82  * - Should be derived from Printable?
83  * - Attributes
84  * - Method name -> defined by constructor? Setter? Builder class?
85  * - Parameters -> defined by dictionary? Attributes in a class?
86  * - Body -> Blob
87  *
88  * Proposition:
89  * - ICommunicators classes are discovered through dynamic_cast, eg.: A SetWiFi command is received,
90  * the corresponding receiver method iterate through all ICommunicators and if some are AWiFiCommunicator,
91  * they update the settings. But this exclude having several communicators of same type connected to several different pairs
92  * - Event receivers are specialized classes implementing an interface, and are associated in a map to the corresponding event
93  * - SendingCommand objects are a helper class used to send a command. Predefined command could have specialized classes implementing their action
94  * */
95  }
